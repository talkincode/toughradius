# ToughRADIUS Feature Checklist

Chinese version: [docs/feature-checklist.md](feature-checklist.md)

Development roadmap: [docs/roadmap.md](roadmap.md) (detailed Chinese roadmap: [docs/roadmap.zh.md](roadmap.zh.md))

This document is the feature scope baseline for ToughRADIUS. Future requirements, issues, pull requests, and code changes must first align with the feature IDs in this checklist. If a request cannot be mapped to an existing feature ID, update this checklist with the scope change before implementation starts.

## Maintenance Rules

1. Every new requirement must reference at least one feature ID, such as `TR-F001`.
2. When adding, splitting, retiring, or changing the direction of a feature, update this checklist first with the status, scope, and acceptance boundary.
3. Features should be delivered in MVP-sized increments: each change must be verifiable, rollback-safe, independently usable, and scoped to the smallest useful unit.
4. Product directions outside this checklist are not default development directions. Add them to the checklist first when there is clear agreement to proceed.
5. This checklist describes product capability boundaries. Concrete APIs, structs, and algorithms remain documented by code, tests, and inline comments.

## Status Definitions

| Status | Meaning |
| --- | --- |
| Core baseline | Mainline project capability. Future changes must preserve compatibility and stability first. |
| Implemented | The main implementation loop exists in code and can continue to receive tests, UX improvements, or edge-case hardening. |
| Partially implemented | Code has an entry point or prototype, but it must not be treated as complete production capability by default. |
| Extensible | This is an allowed extension direction, but it must follow existing architecture and test boundaries. |
| Non-goal | This is not a default project direction at this stage. |

## Priority Extension Directions

The following are priority extension directions and their delivery status (authoritative milestones live in [`docs/roadmap.md`](roadmap.md), with the detailed Chinese execution log in [`docs/roadmap.zh.md`](roadmap.zh.md)). Delivered directions have their capability folded into the feature checklist table above and are retained here to trace scope and development boundaries; undelivered directions must still be broken into MVPs, covered by tests, and referenced by feature ID in issues and pull requests before implementation.

| Priority | Related IDs | Extension Direction | Status | Target Scope | Development Boundary |
| --- | --- | --- | --- | --- | --- |
| P1 | TR-F004 | EAP-TLS support | Delivered (M1) | Add EAP-TLS under the existing EAP handler system, including certificate validation, TLS handshake state management, user identity mapping, and explicit failure reasons. | Do not rewrite the EAP coordinator. Deliver the smallest usable authentication chain first, then extend certificate policy, revocation checks, and management-side configuration. |
| P1 | TR-F010 / TR-F012 / TR-F013 | CoA dynamic authorization support | Delivered (M2) | Provide authorization policy triggers in the user management panel, support CoA / Disconnect actions for online users, and record trigger results. | First abstract a backend CoA sending service and audit result model. The frontend should expose only verifiable safe actions and must not assemble arbitrary RADIUS packets. |
| P1 | TR-F007 / TR-F011 / TR-F015 | IPv6 capability enhancement | Delivered (M3) | Improve IPv6 address, IPv6 prefix, and Delegated-IPv6-Prefix support in users, online sessions, accounting records, audit logs, and dashboard queries/displays. | Do not only add fields to the UI. Protocol parsing, database fields, filters, frontend lists, and audit semantics must form a complete loop. |
| P1 | TR-F004 | PEAPv0 / EAP-MSCHAPv2 | Delivered (M8) | Establish a PEAP TLS tunnel with a server certificate, run EAP-MSCHAPv2 inside it for Windows / AD / legacy enterprise compatibility, and export MPPE session keys correctly. | Compatibility first, not a security selling point. Docs and config must state that MS-CHAPv2 has an NTLMv1-like attack surface (per Microsoft docs); do not weaken the outer TLS by default; do not rewrite the EAP coordinator. |
| P1 | TR-F004 | EAP-TTLS (tunnel + inner PAP/CHAP/MS-CHAP/MS-CHAPv2) | Delivered (M9; inner PAP + MS-CHAP-V2; CHAP/MS-CHAP pending) | Per RFC 5281, build a TLS tunnel with a server certificate and carry PAP / CHAP / MS-CHAP / MS-CHAP-V2 (and inner EAP) inside it, so LDAP, legacy account stores, and hybrid clients can onboard without immediately migrating to a certificate system. | Deliver inner methods one by one (PAP first, then MS-CHAP-V2). Backend user-store adaptation goes through the existing auth pipeline; no store branches at the protocol entry point; do not rewrite the EAP coordinator. |
| P2 | TR-F004 | EAP-TLS 1.3 upgrade (RFC 9190) | Planned (M10) | On top of the TLS 1.2 EAP-TLS baseline delivered in M1, support TLS 1.3 handshake and session key derivation per RFC 9190, following the TLS 1.3 derivation rules in RFC 9427. | Keep backward compatibility with TLS 1.2 clients; negotiate before switching; do not break existing CA-chain validation and identity mapping. |
| P3 | TR-F004 | TEAP (tunnel, machine + user chaining) | Planned (M11) | Per RFC 7170 / RFC 9930 (TEAPv1), implement modern tunneled EAP with machine + user chaining and certificate + password combined authentication; use RFC 9427 derivation under TLS 1.3. | Mid/long-term direction; the client ecosystem is thinner than PEAP. Prioritize only when the client environment is fully controlled; do not compete with PEAP / TTLS for first-version resources. |
| P3 | TR-F004 | EAP-PWD (on demand) | Planned (M12) | Per RFC 5931, authenticate with a shared password without issuing per-client certificates; suitable for IoT, embedded, and controlled small-scale devices. | Not a first choice for general enterprise Wi-Fi; pursue on demand and avoid sinking into a maintenance swamp for the sake of protocol completeness. |
| P2 | TR-F025 | Authentication backend extension: LDAP / AD (bind verification, PAP family) | In progress (M14) | Per RFC 4511 / RFC 4513, verify directory-account passwords via the LDAP / Active Directory bind operation as a pluggable PAP-family verification backend in the auth pipeline, so unified-identity (LDAP/AD) accounts can onboard via plain PAP and `EAP-TTLS/PAP` (delivered in M9) — completing the real backend that EAP-TTLS promised ("let LDAP and legacy account stores onboard without migrating to a certificate system"). Source: issue #199. | PAP family only (where the server can obtain the cleartext password): `CHAP / MS-CHAP / MS-CHAPv2 / EAP-MD5 / PEAP-MSCHAPv2` are **physically impossible** because the server needs the cleartext password (or NT-hash) to compute the challenge response while LDAP bind never releases the password; this must be stated explicitly in docs/config/reject logs and must not be disguised as full support. Hook it in after the existing `internal/radiusd/plugins/auth` pipeline; no store branch at the protocol entry, no rewrite of the auth pipeline, no EAP coordinator change. Do not treat it as a full production loop until M14.6 integration acceptance is delivered. |
| P2 | TR-F023 | Bilingual documentation site (mdbook) | Delivered (M13) | Build a Chinese/English bilingual documentation site with mdbook, consolidating README / AGENT / SECURITY / RFC index and linking to the feature checklist and roadmap, with unified navigation, local build, and CI artifact verification. | Docs do not replace code and tests as the source of truth; Chinese and English directory structures stay corresponding; the feature checklist and roadmap remain living docs under `docs/` and are maintained by dedicated skills. |
| P2 | TR-F024 | Go API documentation and comment standard (standard-library style) | Delivered (M4) | Define and apply a godoc / standard-library comment standard: complete exported-identifier comments, package comments, `Example`s, and error/concurrency semantics; provide a companion skill and measurable gates. | The standard is now part of the lint / CI ratchet. New or changed exported APIs must continue to satisfy standard-library godoc style. |

## Current Non-Goal Directions

| ID | Non-Goal Direction | Explanation |
| --- | --- | --- |
| TR-N001 | Billing payment / orders / finance system | The project currently maintains RADIUS accounting records and audit data only. It should not expand into a full charging system by default. |
| TR-N002 | CRM / ticketing / customer self-service portal | The current management UI is for RADIUS operations management and should not expand into a customer operations platform by default. |
| TR-N003 | General-purpose observability platform | The current dashboard serves RADIUS operations views only and does not replace Prometheus, Grafana, or other general monitoring systems. |
| TR-N004 | Multi-tenant SaaS platform | The current baseline is single-instance management. Multi-tenancy requires a prior design for permissions, data isolation, and migration. |
| TR-N005 | Rewriting the protocol stack or replacing the management framework | Unless there is a clear defect and migration plan, rewriting is not a development direction. |
