# ToughRADIUS Feature Checklist

Chinese version: [docs/feature-checklist.md](feature-checklist.md)

This document is the feature scope baseline for ToughRADIUS. Future requirements, issues, pull requests, and code changes must first align with the feature IDs in this checklist. If a request cannot be mapped to an existing feature ID, update this checklist with the scope change before implementation starts.

## Maintenance Rules

1. Every new requirement must reference at least one feature ID, such as `TR-F001`.
2. When adding, splitting, retiring, or changing the direction of a feature, update this checklist first with the status, scope, and acceptance boundary.
3. Features should be delivered in MVP-sized increments: each change must be verifiable, rollback-safe, independently usable, and scoped to the smallest useful unit.
4. Product directions outside this checklist are not default development directions. Add them to the checklist first when there is clear agreement to proceed.
5. This checklist describes product capability boundaries. Concrete APIs, structs, and algorithms remain documented by code, tests, and inline comments.

## Status Definitions

| Status | Meaning |
| --- | --- |
| Core baseline | Mainline project capability. Future changes must preserve compatibility and stability first. |
| Implemented | The main implementation loop exists in code and can continue to receive tests, UX improvements, or edge-case hardening. |
| Partially implemented | Code has an entry point or prototype, but it must not be treated as complete production capability by default. |
| Extensible | This is an allowed extension direction, but it must follow existing architecture and test boundaries. |
| Non-goal | This is not a default project direction at this stage. |

## Priority Extension Directions

The following directions are allowed to be prioritized as product extensions. Before implementation, each must still be broken into MVPs, covered by tests, and referenced by feature ID in issues and pull requests.

| Priority | Related IDs | Extension Direction | Target Scope | Development Boundary |
| --- | --- | --- | --- | --- |
| P1 | TR-F004 | EAP-TLS support | Add EAP-TLS under the existing EAP handler system, including certificate validation, TLS handshake state management, user identity mapping, and explicit failure reasons. | Do not rewrite the EAP coordinator. Deliver the smallest usable authentication chain first, then extend certificate policy, revocation checks, and management-side configuration. |
| P1 | TR-F010 / TR-F012 / TR-F013 | CoA dynamic authorization support | Provide authorization policy triggers in the user management panel, support CoA / Disconnect actions for online users, and record trigger results. | First abstract a backend CoA sending service and audit result model. The frontend should expose only verifiable safe actions and must not assemble arbitrary RADIUS packets. |
| P1 | TR-F007 / TR-F011 / TR-F015 | IPv6 capability enhancement | Improve IPv6 address, IPv6 prefix, and Delegated-IPv6-Prefix support in users, online sessions, accounting records, audit logs, and dashboard queries/displays. | Do not only add fields to the UI. Protocol parsing, database fields, filters, frontend lists, and audit semantics must form a complete loop. |

## Current Non-Goal Directions

| ID | Non-Goal Direction | Explanation |
| --- | --- | --- |
| TR-N001 | Billing payment / orders / finance system | The project currently maintains RADIUS accounting records and audit data only. It should not expand into a full charging system by default. |
| TR-N002 | CRM / ticketing / customer self-service portal | The current management UI is for RADIUS operations management and should not expand into a customer operations platform by default. |
| TR-N003 | General-purpose observability platform | The current dashboard serves RADIUS operations views only and does not replace Prometheus, Grafana, or other general monitoring systems. |
| TR-N004 | Multi-tenant SaaS platform | The current baseline is single-instance management. Multi-tenancy requires a prior design for permissions, data isolation, and migration. |
| TR-N005 | Rewriting the protocol stack or replacing the management framework | Unless there is a clear defect and migration plan, rewriting is not a development direction. |
